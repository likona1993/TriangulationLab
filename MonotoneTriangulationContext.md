## Существующие типы (в других заголовках)

- `Point2<T>` — поля `x`, `y`; есть `operator-`, метод `.cross()`
- `Polygon2<T>` — `size()`, `operator[]`
- `Triangle2<T>`, `TriangulationResult<T>` (есть поле `error_message`), `DebugStep<T>`
- `Edge` — агрегат `{from, to}` типа `size_t`
- `EPSILON<T>`
- `isSimplePolygon(poly, eps)`
- `isConvex(prev, curr, next, eps)` → `cross(curr-prev, next-curr) > eps`
  (строго левый поворот)
- `computeXIntersection(p1, p2, currentY)` — с защитой от деления на 0 и
  зажимом `t ∈ [0,1]`
- Функция нормализации ориентации в CCW **существует** и вызывается до алгоритма

## Что уже разобрано и решено

1. **Ориентация** — полигон приводится к CCW перед работой. Готово.

2. **Порядок sweep line — лексикографический**: y убывает, при равной y — x
   возрастает. Эквивалентно чуть наклонённой sweep line (`y - εx = const`).
   Благодаря этому нет «одновременных» вершин и нет особых случаев для
   горизонтальных рёбер.

   ```cpp
   static bool isAbove(const Point2<T>& a, const Point2<T>& b) {
     if (std::abs(a.y - b.y) > EPSILON<T>) return a.y > b.y;
     return a.x < b.x;
   }
   ```

   **Критично:** сортировка вершин и `classifyVertex` обязаны использовать
   ОДИН И ТОТ ЖЕ предикат `isAbove`.

3. **`classifyVertex`** — исправлен. Раньше `convex` вычислялся, но не
   использовался, из-за чего SPLIT и MERGE не возвращались никогда.

   | соседи | выпуклая | вогнутая |
   |---|---|---|
   | оба ниже (лок. максимум) | START | SPLIT |
   | оба выше (лок. минимум) | END | MERGE |
   | один выше, один ниже | REGULAR | REGULAR |

4. **`HelperMap`** — был `std::map<Edge, VertexIndex, EdgeCmp<T>>`, это UB:
   компаратор зависит от `currentY`, порядок ключей менялся по ходу sweep и
   дерево разваливалось. Заменён на статический компаратор:

   ```cpp
   struct EdgeLess {
     bool operator()(const Edge &a, const Edge &b) const {
       if (a.from != b.from) return a.from < b.from;
       return a.to < b.to;
     }
   };
   using HelperMap = std::map<Edge, VertexIndex, EdgeLess>;
   ```

5. **`StatusSet`** — компаратор `EdgeCmp<T>` (по x пересечения с sweep line)
   корректен, но только при инварианте: **в статусе лежат ТОЛЬКО нисходящие
   рёбра** (при CCW — рёбра с внутренностью справа). Тай-брейк при почти равных
   x — через `EdgeLess`.

   Ребро `e(a→b)` нисходящее ⟺ `isAbove(poly[a], poly[b])`.
   Вставляется, когда sweep проходит его верхний конец; удаляется — когда нижний.

6. **`computeXIntersection`** для горизонтального ребра должен возвращать
   **левый конец** (`std::min(p1.x, p2.x)`), а не середину — иначе горизонталь
   может оказаться «правее» вершины и выпасть из `findLeftEdge`.

7. **`findLeftEdge`** — статус уже отсортирован по x, поэтому идём слева направо
   и выходим, как только перескочили вершину:

   ```cpp
   auto best = status.end();
   for (auto it = status.begin(); it != status.end(); ++it) {
     const T xEdge = computeXIntersection(poly[it->from], poly[it->to], currentY);
     if (xEdge > v.x + EPSILON<T>) break;
     best = it;
   }
   return best;
   ```
   (O(n) на вершину → O(n²) в худшем случае; TODO — заменить на дерево)

## Ключевая идея фазы 1

**helper(e)** — самая нижняя из уже обработанных вершин, до которой из области
справа от ребра `e` можно безопасно провести диагональ. «Потолок трапеции».

Главная асимметрия:

- **SPLIT** чинится **сразу**: партнёр (helper левого ребра) уже обработан, он выше.
- **MERGE** чинится **отложенно**: партнёр ещё ниже sweep line, мы его не знаем.
  Поэтому MERGE-вершина записывается в helper ребра — это **долг**. Первая же
  вершина, которая позже этого helper'а коснётся, обязана провести к ней диагональ.

Отсюда проверка `types[helper] == MERGE` повторяется в END, MERGE и обеих ветках
REGULAR — это не разные правила, а разные моменты погашения одного долга.

## Хендлеры (текущая версия)

Три кирпичика:

- `closeEdge(vi)` — своё нисходящее ребро `e_prev` кончилось: `tryCloseMerge`,
  `status.erase`, **`helpers.erase`** (не забыть!)
- `openEdge(vi)` — своё нисходящее ребро `e_next` началось: `status.insert`,
  `helpers[e] = vi`
- `updateLeftHelper(vi)` — `findLeftEdge`, `tryCloseMerge`, `helpers[*ej] = vi`

Сводная таблица:

| | closeEdge | openEdge | updateLeftHelper | безусл. диагональ |
|---|---|---|---|---|
| START | | ✓ | | |
| END | ✓ | | | |
| SPLIT | | ✓ | ✓ | **✓** к helper левого ребра |
| MERGE | ✓ | | ✓ | |
| REGULAR, левая цепь (`prev` выше) | ✓ | ✓ | | |
| REGULAR, правая цепь | | | ✓ | |

Состояние sweep передаётся через структуру `SweepCtx` (указатели на poly, types,
status, helpers, diagonals, `T* currentY`). `currentY` — ОДИН экземпляр, на
который смотрит и `EdgeCmp`, и `findLeftEdge`. Обновляется **до** вызова
хендлера (иначе `status.erase` уйдёт не в ту ветку дерева).

## Тестовый пример

Прямоугольник 10×10 с треугольными вырезами сверху и снизу.
Обход CCW:

```
v0=(0,0)  v1=(3,0)  v2=(5,4)  v3=(7,0)  v4=(10,0)
v5=(10,10) v6=(7,10) v7=(5,6) v8=(3,10) v9=(0,10)
```

```
      9┌────────┐8        6┌────────┐5     y=10
       │         ╲        ╱         │
       │          ╲      ╱          │
       │            7 (MERGE)       │      y=6
       │                            │
       │            2 (SPLIT)       │      y=4
       │          ╱    ╲            │
      0└────────┘1      3└────────┘4      y=0
```

Ожидаемые значения:

- **sweep-порядок:** `9, 8, 6, 5, 7, 2, 0, 1, 3, 4`
- **типы (по индексам 0..9):** REGULAR, END, SPLIT, REGULAR, END, REGULAR, START, MERGE, REGULAR, START
- **нисходящие рёбра (только они попадают в статус):**
  `e(9→0)`, `e(6→7)`, `e(2→3)`, `e(0→1)`, `e(3→4)`
- **диагонали:** ровно одна — `{2, 7}`, отрезок (5,4)–(5,6)

Полезно: `v9` — это START, а не REGULAR, потому что при лексикографическом
порядке `v8=(3,10)` считается **ниже** `v9=(0,10)` (та же y, но x больше).
Симметрично `v6` — START, а `v5` — REGULAR.

Инварианты для отладки:
- в конце `status.empty() && helpers.empty()`
- всегда `status.size() == helpers.size()`
- `|status|` = число интервалов сечения (растёт на START/SPLIT, падает на END/MERGE)
